React-formulier

Oorspronkelijk was React niet het plan om op te nemen in mijn iTalent, maar omdat ik toch graag nog gebruik wilde maken van een formulier zodat ik zelf data kon voorzien voor mijn Mule-applicatie, heb ik gekozen voor React, omdat ik hier nog geen ervaring mee had.

Het framework React werd gebruik voor het formulier op te stellen. (Bekijk de volledige demo in een filmpje bij het Mule-gedeelte). Deze technologie behoorde oorspronkelijk niet tot een van de actitiveiten die ik voor ogen had voor iTalent, maar toen ik besloten had om een formulier te voorzien, wilde ik hier ook graag een nieuwe technologie voor gebruiken, wat uiteindelijk React geworden is.

Ik ben vrijwel onmiddellijk begonnen aan mijn formulier, zonder een tutorial te hebben gekeken. Wel hield ik ook altijd de documentatie open van React zelf. Ik ben begonnen door mij eerst te focussen op een formulier volledig gemaakt in HTML, zodat ik de juiste velden al had. Vervolgens ben ik officieel begonnen aan de React-app en heb ik deze gegenereerd via de terminal. Ik heb een view aangemaakt voor het formulier en vervolgens heb ik de HTML omgezet in JSX, wat React gebruikt voor het UI-gedeelte. Het kan een beetje vergeleken worden met HTML gemixt met JavaScript.

Verder heb ik ook gebruik gemaakt van een aantal hooks van het React-framework. Hooks zijn een aantal standaardfuncties van React die je kan gebruiken om bijvoorbeeld “in te haken” op de state van je data. Zo heb ik bijvoorbeeld de hook useState gebruikt, om een object en hiervan de initiële state te ‘setten’, en daarbij ook een functie te voorzien om deze state te kunnen overschrijven.

Aangezien er op het formulier een aantal zaken meermaals terugkomen, heb ik daar gebruik gemaakt van componenten. Ik heb een component voor een input field gemaakt, en een component voor radio buttons.

Met de data die wordt ingegeven in het formulier, moet ook iets gebeuren, namelijk verstuurd worden naar de Mule-applicatie zodat dat verder afgehandeld kan worden. Wanneer er op de submit-knop geklikt wordt, wordt er eerst gevalideerd of er iets is ingevuld in alle verplichte velden, en of de ingevulde waarden correct zijn. Vervolgens steek ik de waarden ik een data-object, dat ik meegeef via een POST-request naar mijn Mule-applicatie. Het POST-request stuurt ook een resultaat terug, wat ik vervolgens heb opgeslagen in de session storage. Zo blijft de data de hele sessie opgeslagen, maar zodra de sessie beëindigd is, zal deze data ook verwijderd worden.

Als ik de status 200 terugkrijg, zorg ik er ook voor dat ik naar de view van de resultaten geleid wordt. Dit doe ik met de navigate-hook van React Router. Op deze manier zorg ik ervoor dat de state van componenten bewaard blijft. In deze POC heeft dat niet heel veel meerwaarde, omdat ik in de view van de resultaten data binnenkrijg van de Mule-applicatie en niet uit de React-app, maar ik wilde er toch een best practice van maken door toch gebruik te maken van de Router.

Het resultaten-scherm heb ik heel eenvoudig gehouden. Ik haal de data uit de session storage op en toon vervolgens de namen van de steden op het scherm. Voor gedetailleerde informatie van de steden, heb ik gebruik gemaakt van een e-mailfunctionaliteit, waarover later meer.

Reflectie

Door mijn basiservaring in Vue was ik redelijk snel weg met React. Ik besef heel goed dat dit een POC is en geen uitgebreide applicatie, waarbij ook geen state management zoals Redux werd toegepast.

Ik ben niet echt tegen problemen aangelopen, maar er waren een aantal zaken die ik extra heb moeten opzoeken om ze volledig te kunnen begrijpen, zoals de hooks die ik gebruikt heb. In de opleiding hebben we al Vue en Angular aangeraakt gehad, maar nog geen React, waardoor ik deze technologie wilde gebruiken om mijn kennis wat uit te breiden.

Ik heb bewust geen tijd gestoken in het extra stijlen van het formulier, omdat er al vrij veel tijd stak in de Mule-applicatie. Ik heb mij hier volledig gefocust op het praktische, en minder op het esthetische.

Mijn sterke punten waren dat ik vrij vlot weg was met de technologie. Dit kwam voornamelijk door al enige ervaring in het werkveld op vlak van frontend. Daar bovenop was de documentatie van React vrij duidelijk, wat mij hielp om snel op weg te geraken. Indien er toch iets niet duidelijk was, had ik even de tijd genomen om het even door te lezen op Stack Overflow of een blogbericht.

Een zwak punt bij mij is dat ik vrij snel gefrustreerd raak als iets niet onmiddellijk lukt bij frontend-technologieën. Als er bijvoorbeeld bij bepaalde properties een typfout is gemaakt of er zit ergens een logicafout in een loop, wordt er geen error getoond op het scherm. Er wordt dan volledig niets getoond. Ook debuggen is niet zo eenvoudig, waardoor ik al snel aan een aantal console.logs zit en zo steeds moet proberen te ontcijferen waar de fout exact zit. Ik merk dat ik daar niet zo veel geduld voor heb.

Ik heb deze opdracht gekozen voor de bespreking in het portfolio omdat dit een onderdeel vormt van mijn grote Mule-opdracht. Ik heb hiervoor speciaal gekozen voor een nieuwe technologie, om mij ook op dit vlak nog wat bij te scholen.
